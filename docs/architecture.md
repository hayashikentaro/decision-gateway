# Decision Gateway Architecture

This document describes the intended architecture for Decision Gateway. It is a product and responsibility map, not a claim that all pieces are implemented.

## Product Invariant

Decision Gateway owns the human judgment UX. Sources ask for decisions; Decision Gateway creates the Decision Workspace where the human reviews context and records judgment.

TaskDeck may be the first source connector, but Decision Gateway is not a TaskDeck feature and should not expose the TaskDeck server directly.

## Main Flow

```text
Source system
  -> Decision Request ingestion
  -> Decision record
  -> Decision Workspace
  -> Notification connector
  -> Human opens workspace and decides
  -> Decision Result record
  -> Future return path
```

## Components

### Decision Request Ingestion

Accepts structured requests from source systems. Ingestion validates that the request has a source, goal, decision axis, urgency, decision question, summary, materials, and enough context for a human to judge.

TaskDeck is expected to be the first source connector, but the request protocol must remain source-neutral.

### Decision Workspace

The workspace is the primary decision surface. It should present:

- the decision question;
- source and goal context;
- decision axis and urgency;
- semantic summary;
- materials and links;
- recommended decision when provided;
- stale-state warnings;
- actions for decision, instruction, or insufficient materials.

The workspace is owned by Decision Gateway. Source systems may provide context and materials, but they do not generate the workspace UI.

### Notification Connector

The notification connector sends an entry-point message to the human with a link to the Decision Workspace. Notification payloads should be minimal and should not become the decision surface.

The connector should avoid spam by requiring a clear decision question, respecting urgency, and supporting dedupe or replacement behavior when a request becomes stale.

### Decision Result

Decision Gateway records the human decision and any agent instruction. Result delivery back to the source system is a future protocol capability, not an MVP requirement.

### Cloud Mailbox Future Return Path

A future return path may use a cloud mailbox or similar durable delivery mechanism so source systems can retrieve decision results without direct server exposure.

This should support target addressing, delivery mode, stale handling, and retry semantics without binding the protocol to TaskDeck.

## TaskDeck Relationship

TaskDeck is a connector/source/orchestration host. It can emit decision requests when agent work needs human judgment. Decision Gateway then owns notification, workspace, decision capture, and future result delivery.

## Non-Goals

- Direct TaskDeck server exposure.
- TaskDeck-only request or result protocols.
- Notification-only approval workflows.
- Decision Workspace UI generated by TaskDeck.
- General agent orchestration or connector hosting in this repository.
